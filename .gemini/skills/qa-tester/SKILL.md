# Testing Skill - Repository: URL-Shortener

Purpose
-------
Provide a reusable, repo-scoped skill that enforces and automates the test requirements for every feature in this project. The skill guides engineers and automation to ensure: unit tests, integration tests, edge-case generation, validation tests, Mockc coverage for controllers, Thymeleaf view checks, documentation and Swagger/API schema generation, and quality gates (linting, security, perf).

When to use
-----------
- Before merging a change that adds or modifies a feature
- As part of CI job that validates PRS
-When adding new capabilities (APIS, UI, background jobs) 

High-level workflow
----------------
1. Identify feature scope (API, UI, DB, events).
2. Require a minimal test matrix for the feature:
  - Unit tests for all business logic classes
  - Integration tests that exercise persistence and wiring
  - Controller/API tests (Mockvc) for request/response and validation
  - VI smoke tests for Thymeleaf views (rendering + common flows)
  - validation tests for invalid inputs and error branches
  - Edge-case test generation: add tests for boundary conditions, nulls, long inputs, concurrency where relevant
3. Ensure API surface has schema/swagger and documentation updates.
4. Run quality gates: linting, static security scans, basic performance smoke tests (optional)
5. Submit change with results and test coverage notes; CI must pass before merge.

Decision points and branching
-----------------------------
- If feature touches only internal logic (no API/UI), skip MockMvc and Thymeleaf tests but keep unit + integration.
- If feature introduces new external dependency (Redis, Kafka, external API), prefer integration tests that can run in Docker (CI) or use testcontainers.
- If tests are flaky, add a clear plan to stabilize (timeouts, deterministic seeds) and mark flakiness in PR.


Quality criteria / completion checks
------------------------------------
- Unit tests: cover public methods and validation logic for the feature.
- Integration tests: exercise repository/service wiring and fail on missing beans or config.
- Mockvc tests: status codes, validation messages, headers, and JSON schema shape for APIs.
- Thymeleaf: pages render without errors and include expected form fields and links.
- Edge cases: include at least 3 negative tests for each critical input path.
- Documentation: update README, API docs, and change proposal if feature scope changed.
- Swagger/API schema: endpoints added to autogenerated openAPI artifact (or PR includes schema file).
- Quality gates: lint (vn fat/checkstyle), security scan (OWASP dependency check or SCA), and a perf smoke (simple timing assertion) where relevant.

Suggested tests matrix (minimal per feature)
--------------------------------------------
- Unit: 80% of newly added service/repository/utility code.
- Integration: 1-3 tests covering main happy path and 1 error path.
- MockMvc: POST create, GET fetch, PUT update, DELETE remove flows and validation error cases.
- UI (Thymeleaf): render dashboard, create form submit flow (happy + validation).

Test-case generation guidance
-----------------------------
-Unit: 80% of newly added service/repository/utility code.
- Integration: 1-3 tests covering main happy path and 1 error path.
- MockMvc: POST create, GET fetch, PUT update, DELETE remove flows and validation error cases.
- UI (Thymeleaf): render dashboard, create form submit flow (happy + validation).

Test-case generation guidance
-----------------------------
- For inputs, generate:
  - Valid baseline case
  - Invalid formats (e.g., malformed URLS)
  -Boundary lengths (empty, max length + 1)
  - Concurrency/duplicate alias case
- For each generated case, assert both behavior and appropriate error messages or status codes.

Automation / CI recommendations
- Run /mvnw test* and /mvw -DskipTests-false package* in CI.
- Use Testcontainers for PostgresQL/Redis/Kafka integration tests in CI.
- Fail the pipeline on:
  - Test failures
  - New high-severity SCA findings
  - Checkstyle/format violations
